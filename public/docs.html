transparency

How trench works.

Every part is public: the pool wallet, the exact schedule, who gets paid, and the bot that runs it. Nothing happens off-chain or off the books.

01
the flow

From order to drop.

1 · Open an order

A dev pastes their token CA, chooses how much supply to drop, a target market cap, and the round shape. The site shows the exact schedule before anything is sent.

2 · Fund the pool

They send that supply to one public pool wallet with a unique memo code. The bot matches the deposit by memo + token mint and marks the order funded.

3 · Drops fire

When the market cap hits each round's threshold, the bot snapshots holders and sends an even cut. Status updates show live on the pool.

02
the pool wallet

One public address.

All locked supply sits in a single public wallet. Every deposit and every drop is an on-chain transaction anyone can verify on a block explorer.

03
the rounds

A schedule, not a promise.

Each order picks a number of rounds, a percent per round, and how the cap climbs. The first drop fires at the target cap; the next at each ×2 (or +step) after. Defaults: 5 rounds, 50% of the remaining balance, ×2 cap — but it's fully configurable (e.g. 10×10%, 5×20%, 20×5% gentle). The bot runs the exact schedule shown on the site when the order was created.

04
the holders

Who gets paid.

Each round takes a fresh snapshot of holders ranked #1 to #1000 (configurable per order) and splits the round's amount evenly — a rank-1 wallet and a rank-1000 wallet receive the same. The liquidity pool is always excluded, plus any team/CEX/market-maker addresses the dev lists, so the drop never just recycles back into the LP.

05
the bot

Open source, no hidden logic.

The bot that watches deposits, snapshots holders and sends the drops is fully public — you can read every line. It writes all status changes through the same API the live pool uses, so what you see on the site is what actually ran on-chain.
